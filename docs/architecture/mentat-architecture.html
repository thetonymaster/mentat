Mentat · Trace Behaviour Test Framework

architecture · 2026-06-17 · working draft
As of phase
Path

Assert how a system behaved — and what it produced — by inspecting its OpenTelemetry trace. Write Gherkin, drive the system, fetch the trace from Tempo, and run comparators that encode "correct behaviour." The comparator core is the stable center; the trace source is a swappable adapter, so the same checks run against AI/LLM agents and microservices alike. Toggle the controls above to see the system at each phase, or per target path.

Why it's portable — one core, two paths

Agent path
mentatctl shell/mcp → agent run. Output = final answer. Trace = invoke_agent / execute_tool spans.
Microservice path
http/grpc → request. Output = response body+status. Trace = service spans. traceparent propagation is native here.
▶
Evidence
{ Trace, Output }
the single currency every comparator consumes
▶
Comparator core
Same sequence / budgets / result / shape comparators run on either path. They don't know who fetched the trace or drove the SUT — that's the whole design.

Microservices exercise 3 of 4 comparators heavily (sequence, shape, budgets) plus deterministic result; the novel/risky semantic matcher is the one they don't need.

Correlation — baggage-first

Baggage is the primary mechanism: it survives the agent starting its own root trace, and lands a queryable tag on every span. traceparent is a complement (a clean single trace ID when the SUT honors it).

1
Inject baggage
Driver sets test.run.id=<uuid> (+ scenario/case) as W3C baggage — `baggage` header for http/grpc, env for shell/mcp. Also mints traceparent.
2
SUT propagates
SUT's W3C baggage propagator carries it through every downstream hop — even across new traces the agent starts itself.
3
Stamp onto spans
A BaggageSpanProcessor in the SUT copies test.run.id onto every span as an attribute → queryable.
4
Resolve + stable-poll
TraceQL { test.run.id = "…" } → trace(s); poll Tempo until span count is stable (ingestion lag).
!
No silent fallback
Not-found / unexpected multi-trace = hard, descriptive failure. Never guess.

Integration contract (the one thing we require of the SUT): honor the W3C baggage propagator + run a baggage→span-attribute processor. OTel ships these for major languages. Our fakeagent fixture does exactly this, so the hermetic E2E exercises the real baggage path.

Core contracts

// every seam is an interface; impls are injected at the composition root. type Driver interface { Run(ctx, RunSpec) (RunResult, error) } type TraceStore interface { GetByID(ctx, id) (*Trace, error); Query(ctx, TraceQuery) ([]TraceRef, error); Caps() StoreCaps } type Correlator interface { Inject(ctx, *RunSpec) (runID string); Resolve(ctx, TraceStore, runID) (*Trace, error) } type Comparator interface { Compare(ctx, Evidence, Expectation) (Verdict, error) } type Matcher interface { Match(got, want Value) (Verdict, error) } // inside result comparator type Judge interface { Judge(ctx, got, criteria string) (Verdict, error) } // semantic matcher → Claude type Reporter interface { Report(ScenarioResult) error } type RunResult struct { TraceID string; Output Output } // trace ID + boundary output type Evidence struct { Trace *Trace; Output Output } // what a comparator sees type Verdict struct { Pass bool; Reasons []string } // reasons → report

Extensibility — dependency injection & seams

Composition root — engine.Build(cfg)
Reads mentat.yaml, resolves each dependency from its registry by name, and constructor-injects the chosen implementations into the Engine. The Engine depends only on the interfaces above — never on a concrete Tempo, shell adapter, or Claude. Idiomatic Go DI: interfaces + constructor injection + a registry per seam (no framework; google/wire optional for compile-time wiring).

Adding a capability = implement the interface + register a factory. No engine change. The otlp-file / inmem stores aren't just future-proofing — they let us unit/E2E test comparators with zero infrastructure. That's the pluggability paying rent immediately.

Phasing — each a verified checkpoint

v1 Thin E2E

  • godog → shell adapter
  • correlate + stable-poll
  • sequence + budgets + deterministic result
  • JUnit; hermetic green/red proof

P2 Portability

  • http adapter
  • microservice through the same comparators
  • response body/status result
  • proves swappable source

P3 Shape + span results

  • shape comparator (TraceQL)
  • sidecar YAML expectations
  • span-attribute result source
  • per-tool / intermediate results

P4 Semantic

  • LLM-judge result matcher (Claude)
  • pluggable Judge
  • @runs(N) non-determinism
  • pass policy: all / majority

P5 Breadth

  • grpc / mcp adapters
  • optional tracetest adapter
  • wider SUT coverage
